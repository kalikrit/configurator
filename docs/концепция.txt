# Конфигуратор устройств — концепция архитектуры

## 1. Общее описание

Конфигуратор позволяет пользователю собирать сложные устройства (например, персональный компьютер) из отдельных компонентов. Процесс сборки имитирует реальный процесс выбора комплектующих: пользователь выбирает компоненты в произвольном порядке, система контролирует их совместимость и накапливает итоговые характеристики.

Ключевая особенность — **универсальность**: архитектура позволяет добавлять новые типы устройств (серверы, рабочие станции, ноутбуки и т.д.) без изменения кода, за счёт описания устройства через слоты и правила совместимости.

---

## 2. Основные сущности

### 2.1. Устройство (Device)
Сущность, которую собирает пользователь. Определяет:
- Набор слотов (см. ниже)
- Правила завершённости сборки (какие слоты обязательны)
- Глобальные ограничения (например, максимальное энергопотребление)

Пример: `Персональный компьютер`, `Сервер`, `Рабочая станция`.

### 2.2. Слот (Slot)
Место в устройстве, которое может быть заполнено компонентом. Характеристики слота:
- Уникальный идентификатор
- Название (отображается пользователю)
- Категория компонентов, которые могут быть установлены
- Количество (1 или несколько, например 4 слота для памяти)
- Обязательность (должен быть заполнен для завершения сборки)
- Зависимости от других слотов (например, тип памяти определяется выбранной материнской платой)

### 2.3. Компонент (Component)
Конкретная комплектующая из каталога. Содержит:
- Базовую информацию: название, производитель, цена, изображение
- Технические характеристики (сокет, форм-фактор, объём, частота, мощность и т.д.)
- Правила совместимости (с какими другими компонентами может работать)

### 2.4. Сборка (Build)
Состояние, создаваемое пользователем. Содержит:
- Выбранное устройство
- Список выбранных компонентов с привязкой к слотам (для множественных слотов — массив)
- Итоговые вычисленные характеристики (суммарная стоимость, общий объём памяти и т.д.)
- Статус сборки (черновик, завершена)

### 2.5. Каталог (Catalog)
Хранилище всех доступных компонентов с их характеристиками. На первом этапе — статический JSON, в перспективе — API.

## 3. Правила совместимости

Совместимость проверяется по двум направлениям:

1. **Компонент → Слот**  
   Компонент должен подходить в слот по своей категории и техническим характеристикам (например, процессор должен иметь сокет, поддерживаемый материнской платой).

2. **Компонент ↔ Компонент**  
   Компоненты должны быть совместимы друг с другом (например, материнская плата и процессор должны иметь одинаковый сокет).

Проверки выполняются **при каждом добавлении или изменении компонента**:

- Новый компонент проверяется на совместимость со **всеми уже выбранными** компонентами
- Если несовместим — пользователь получает сообщение с объяснением причины, добавление блокируется
- При замене существующего компонента — аналогичная проверка

### 3.1. Типы проверок совместимости

| Связь | Проверяемые параметры |
|-------|----------------------|
| Процессор ↔ Материнская плата | Сокет, TDP (для выбора охлаждения) |
| Материнская плата ↔ Память | Тип памяти (DDR4/DDR5), частота (предупреждение), максимальный объём, количество слотов |
| Материнская плата ↔ Накопитель | Тип разъёма (M.2, SATA), количество доступных слотов |
| Материнская плата ↔ Видеокарта | Версия PCIe (обратная совместимость), длина (с учётом корпуса) |
| Видеокарта ↔ Блок питания | Минимальная мощность, наличие необходимых разъёмов |
| Корпус ↔ Материнская плата | Форм-фактор (ATX, mATX, Mini-ITX) |
| Корпус ↔ Видеокарта | Максимальная длина видеокарты |
| Корпус ↔ Кулер | Максимальная высота кулера |
| Кулер ↔ Процессор | Поддержка сокета, достаточная мощность охлаждения (TDP) |
| Блок питания ↔ Все потребители | Суммарное расчётное энергопотребление ≤ мощность БП |

### 3.2. Множественные компоненты (память, накопители)

- Пользователь может добавлять несколько планок памяти / накопителей, пока не будут заполнены все доступные слоты
- При добавлении нового экземпляра проверяется совместимость с уже установленными (не смешивать DDR4 и DDR5, желательно одинаковые частоты — предупреждение)
- Система суммирует объёмы и считает итоговые характеристики

## 4. Логика работы пользователя

### 4.1. Начало работы

1. Пользователь открывает конфигуратор
2. Выбирает тип устройства (если поддерживается несколько типов)
3. Открывается пустая сборка, левая панель показывает все категории компонентов

### 4.2. Добавление компонентов

1. Пользователь выбирает категорию (процессоры, материнские платы и т.д.)
2. Видит список компонентов с характеристиками и ценами
3. Несовместимые с уже выбранными компонентами варианты:
   - Визуально затемнены
   - При наведении показывают причину несовместимости
   - Недоступны для выбора
4. При выборе компонента:
   - Проверяется совместимость со всеми уже выбранными
   - Если совместим — добавляется в сборку
   - Если нет — показывается сообщение об ошибке, добавление блокируется

### 4.3. Управление сборкой

- Пользователь может заменить любой выбранный компонент на другой из той же категории
- При замене материнской платы (центрального компонента) система проверяет совместимость с остальными выбранными компонентами:
  - Если есть несовместимые — предлагает либо отменить замену, либо удалить несовместимые компоненты
- Пользователь может удалить любой компонент из сборки
- При удалении компонента система пересчитывает совместимость оставшихся

### 4.4. Завершение сборки

Сборка считается завершённой, когда:
- Заполнены все обязательные слоты (определяется устройством)
- Отсутствуют предупреждения о несовместимости (или они не критические)

После завершения (или в процессе) пользователь видит:
- Итоговую стоимость
- Суммарные характеристики (объём памяти, объём хранилища, расчётное энергопотребление)
- Список выбранных компонентов
- Кнопки сохранения / загрузки конфигурации

---

## 5. Итоговые вычисления (агрегация)

Система автоматически пересчитывает следующие метрики при любом изменении сборки:

| Параметр | Способ расчёта |
|----------|---------------|
| Общая стоимость | Сумма цен всех выбранных компонентов |
| Общий объём памяти | Сумма объёмов всех выбранных планок памяти |
| Общий объём хранилища | Сумма объёмов всех выбранных накопителей |
| Расчётное энергопотребление | Сумма TDP/потребления всех компонентов (CPU, GPU, память, накопители) + запас 10-20% |
| Совместимость | Общий флаг (все проверки пройдены) + список предупреждений |

---

## 6. Сохранение и загрузка конфигураций

- Пользователь может сохранить текущую сборку в локальное хранилище (IndexedDB / localStorage) или на сервер
- Сохранённая конфигурация содержит:
  - Идентификатор устройства
  - Список выбранных компонентов (ID и количество для множественных)
  - Метаданные (дата создания, название, описание)
- При загрузке конфигурации система восстанавливает состояние сборки и проверяет актуальность компонентов (если каталог обновился)

---

## 7. Расширяемость

### 7.1. Добавление нового устройства

1. Создаётся JSON-описание устройства:
   - Название
   - Список слотов с их характеристиками
   - Правила завершённости
2. Устройство добавляется в каталог устройств
3. Конфигуратор автоматически отображает его в списке выбора

### 7.2. Добавление нового компонента

1. В JSON каталога добавляется запись с:
   - Категорией
   - Характеристиками
   - Ценой
2. Компонент автоматически появляется в соответствующей категории левой панели
3. Валидатор использует его характеристики при проверках совместимости

### 7.3. Добавление новых правил совместимости

- Правила реализованы в виде централизованного валидатора
- Добавление нового правила не требует изменения UI или логики сборки
- Правила могут быть общими (для всех устройств) или специфичными для конкретного устройства

## 8. Технические слои (без привязки к фреймворку)

1. **Слой данных**  
   - Каталог компонентов (JSON / API)
   - Каталог устройств (JSON)
   - Состояние сборки (Pinia / Vuex / аналоги)

2. **Слой бизнес-логики**  
   - Валидатор совместимости
   - Агрегатор характеристик
   - Сериализатор / десериализатор конфигураций

3. **Слой представления**  
   - Левая панель (категории, список компонентов с фильтрацией по совместимости)
   - Центральная область (отображение текущей сборки, схематичное или табличное)
   - Правая панель (итоговая информация, стоимость, характеристики)

4. **Слой хранения**  
   - Локальное сохранение (localStorage / IndexedDB)
   - Серверное API (опционально)

---

## 9. Приоритеты разработки (MVP)

1. Реализовать каталог компонентов (статический JSON) для одного устройства (ПК)
2. Реализовать базовую логику совместимости (основные пары: CPU ↔ MB, MB ↔ RAM, MB ↔ Storage, GPU ↔ PSU, Case ↔ MB/GPU)
3. Реализовать UI: выбор компонентов с визуализацией совместимости
4. Реализовать агрегацию (стоимость, суммарные характеристики)
5. Добавить возможность множественного выбора (память, накопители)
6. Реализовать сохранение / загрузку конфигураций (локально)
7. Добавить поддержку нескольких устройств (расширить архитектуру)

## 10. Открытые вопросы (требуют уточнения)

- **Видеокарта:** считать обязательной, если у процессора нет встроенной графики? Или всегда опциональной, но с предупреждением?
- **Кулер:** как моделировать ситуацию, когда процессор идёт с комплектным кулером (box-версия)? Выделять отдельный компонент или считать кулер частью процессора?
- **Завершённость сборки:** определять по наличию всех обязательных слотов или дать пользователю возможность сохранять незавершённые сборки как черновики?
- **Замена центрального компонента (материнской платы):** автоматически удалять несовместимые компоненты или предлагать пользователю выбор?
